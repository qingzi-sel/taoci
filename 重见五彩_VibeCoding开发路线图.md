# 《重见五彩》Vibe Coding 开发路线图

> 版本：v0.1  
> 日期：2026-06-16  
> 用途：把《重见五彩_初版GDD.md》拆成可执行的开发阶段，帮助以 vibe coding + AI 美术的方式逐步做出可玩的游戏。  
> 核心原则：先做可玩闭环，再扩内容；先验证规则，再替换美术；每次只让 AI 完成一个明确功能。

## 1. 当前判断

《重见五彩》的 GDD 方向已经足够清楚，但目标体量偏大，不适合直接让 AI 一次性生成完整游戏。

本项目应采用“小闭环迭代”的制作方式：

> 灰盒可玩 → 规则成立 → 垂直切片 → 扩内容 → 换美术 → 打磨发布

不要从完整经营系统、完整章节、完整角色成长和完整美术资产开始。第一阶段只验证一个核心体验：

> 玩家经历一次失败，理解失败原因，调整工艺方案，再次开窑并获得成功。

## 2. 最小可玩目标

第一版不做完整游戏，只做 10-15 分钟可玩原型。

### 2.1 玩家流程

1. 进入工坊。
2. 接到恢复釉下五彩的任务。
3. 完成一次旧流程作品，发现颜色发暗或周期过长。
4. 尝试二烧制，第一次出现失败。
5. 查看失败复盘，理解问题原因。
6. 调整工艺参数。
7. 再次开窑，获得较好结果。
8. 解锁一条史实档案，结束原型。

### 2.2 第一版只保留的系统

- 工坊界面
- 工艺参数选择
- 开窑结果判定
- 失败复盘
- 档案解锁
- 简单剧情对白

### 2.3 第一版暂不做的内容

- 完整经营系统
- 复杂好感度
- 多章节自由推进
- 大量订单
- 完整图鉴
- 存档系统
- 全角色立绘
- 分支结局
- 真实陶瓷参数模拟

## 3. 总体开发阶段

### 阶段一：执行文档整理

目标：把大 GDD 压缩成可开发的小规格。

#### 产出文件

建议先建立以下 4 个文档：

1. `重见五彩_垂直切片执行稿.md`
2. `重见五彩_失败规则表.md`
3. `重见五彩_界面清单.md`
4. `重见五彩_AI美术资产清单.md`

#### 验收标准

- 能说清楚第一版玩家 10-15 分钟内做什么。
- 能列出第一版所有界面。
- 能列出 5-8 种失败结果。
- 能说清楚玩家如何从失败走向成功。
- 不再继续扩写世界观和长期设定。

### 阶段二：灰盒原型

目标：不用正式美术，先让游戏完整跑通。

#### 必做功能

1. 开始界面。
2. 工坊界面。
3. 参数选择界面。
4. 开窑按钮。
5. 结果界面。
6. 失败复盘界面。
7. 再试一次。
8. 成功后进入结尾。

#### 技术建议

如果目标是快速做出网页试玩版，优先使用：

- HTML
- CSS
- JavaScript

如果后续明确要做桌面游戏，再考虑 Godot 4。但第一版不建议先上复杂工程结构。

#### 验收标准

- 可以从开始玩到结尾。
- 玩家至少失败一次。
- 玩家能根据复盘提示调整参数。
- 第二次或第三次尝试可以成功。
- 没有正式美术也能理解流程。

### 阶段三：核心规则

目标：让失败和成功有明确因果，而不是随机按钮。

#### 初版参数

建议先保留 4 个参数：

| 参数 | 可选值 | 作用 |
| --- | --- | --- |
| 釉料 | 旧石灰青釉 / 长石釉 | 影响底色、发灰概率 |
| 罩釉厚度 | 薄 / 中 / 厚 | 影响保护、排气、针孔和气泡 |
| 升温节奏 | 慢 / 中 / 快 | 影响有机物排出和烧成稳定 |
| 氧化气氛 | 不足 / 普通 / 充分 | 影响黑壳、发乌、呈色稳定 |

#### 初版失败规则

可先使用简单条件判断：

| 条件 | 结果 | 原因 | 建议 |
| --- | --- | --- | --- |
| 旧石灰青釉 | 颜色发灰 | 旧釉底色偏青偏暗，影响釉下色呈现 | 尝试长石釉 |
| 厚罩釉 + 快升温 | 针孔或气泡 | 釉层过厚且升温太快，排气窗口不足 | 降低罩釉厚度，放慢升温 |
| 厚罩釉 + 氧化不足 | 黑壳或发乌 | 有机物排出不充分，且釉层封闭过快 | 使用薄罩釉，提高氧化充分度 |
| 快升温 + 氧化不足 | 发乌 | 烧成过程过急，气氛控制不足 | 放慢升温，增强氧化 |
| 长石釉 + 薄罩釉 + 慢升温 + 氧化充分 | 成功 | 釉面、排气和呈色较稳定 | 进入展柜与档案解锁 |

#### 验收标准

- 每个失败都有明确原因。
- 每个失败都有下一步建议。
- 玩家能通过理解规则提高成功率。
- 成功不是纯随机。

### 阶段四：垂直切片

目标：做出可以给别人试玩的 20-35 分钟版本。

#### 推荐流程

| 时间 | 内容 | 玩家行为 | 目标 |
| --- | --- | --- | --- |
| 0-3 分钟 | 开场 | 阅读短对白，进入工坊 | 理解抢救任务 |
| 3-7 分钟 | 三烧制教学 | 选择旧流程，完成一次作品 | 理解旧工艺能恢复但有负担 |
| 7-10 分钟 | 问题暴露 | 查看作品发灰、周期长、成本高 | 产生改进动机 |
| 10-16 分钟 | 二烧第一次试验 | 选择参数并开窑 | 触发黑壳、针孔或发乌 |
| 16-22 分钟 | 失败复盘 | 阅读原因，调整参数 | 理解二烧不是少烧一次 |
| 22-28 分钟 | 二烧第二次试验 | 使用改进方案再次开窑 | 获得稳定结果 |
| 28-35 分钟 | 展柜结尾 | 展示作品，解锁档案 | 完成主题表达 |

#### 验收标准

- 玩家能独立玩完。
- 玩家能说出釉下五彩不是普通填色。
- 玩家能理解二烧制不是简单少烧一次。
- 玩家能感受到团队抢救，而不是个人发明。
- 结尾有作品展示和档案解锁。

### 阶段五：内容扩展

目标：在不推翻核心系统的情况下，逐步增加内容。

#### 推荐扩展顺序

1. 增加角色对白。
2. 增加失败类型。
3. 增加订单任务。
4. 增加档案图鉴。
5. 增加章节。
6. 增加最终作品选择。
7. 增加分水小游戏。
8. 增加角色成长。

#### 每次扩展的限制

一次只做一个扩展点。

错误做法：

> 同时增加新角色、新章节、新系统、新美术和新规则。

正确做法：

> 只增加“失败类型：留白不清”，完成后测试旧流程仍可玩，再继续下一个功能。

#### 验收标准

- 新内容没有破坏旧流程。
- 从开始到结尾仍然能完整玩通。
- 新内容服务“抢救、试验、复盘、传承”的主题。

### 阶段六：AI 美术替换

目标：用 AI 美术逐步替换占位图，而不是一开始堆大量图片。

#### 推荐资产顺序

1. 工坊背景
2. 窑炉间背景
3. 制作台背景
4. 失败瓷器图
5. 成功瓷器图
6. 最终五彩瓶
7. 角色立绘
8. 档案插图
9. UI 装饰元素

#### 资产记录格式

每张图都记录：

```text
资产名称：
用途：
尺寸：
风格关键词：
生成提示词：
是否已放入游戏：
是否需要重做：
```

#### 美术原则

- 先生成关键场景和结果图。
- 不要一开始生成大量角色。
- 不要让不同图片风格差异过大。
- 图片要帮助玩家理解状态、结果或情绪。
- 最终作品不能直接复刻真实文物。

#### 验收标准

- 每张图都有明确用途。
- 关键流程都有视觉反馈。
- 成功和失败结果能一眼区分。
- 风格统一，不像不同项目拼接。

### 阶段七：打磨发布

目标：让游戏从“能玩”变成“别人愿意玩完”。

#### 打磨清单

- 开场是否太长。
- 教程是否太密。
- 按钮是否清楚。
- 失败原因是否好懂。
- 结果反馈是否有成就感。
- 是否能重新开始。
- 是否存在卡死流程。
- 是否有明显错别字。
- AI 图是否风格统一。
- 档案是否过长。

#### 试玩验证

找 3-5 个人试玩，观察他们是否：

- 不看说明也能开始。
- 知道下一步该点哪里。
- 失败后知道怎么调整。
- 能玩到结尾。
- 能复述游戏核心。

#### 验收标准

- 至少 3 个外部玩家能独立玩完。
- 大多数玩家能理解“二烧不是少烧一次”。
- 大多数玩家能记住釉下五彩、长石釉、分水、开窑。
- 没有阻断流程的 bug。

## 4. 每次 Vibe Coding 的标准流程

每做一个功能，都按以下流程：

1. 写一句目标。
2. 写清楚输入。
3. 写清楚输出。
4. 让 AI 只实现这一件事。
5. 运行测试。
6. 修 bug。
7. 确认旧流程没坏。
8. 保存当前版本。

### 示例：开窑结果判定

目标：

```text
实现开窑结果判定，根据玩家选择的工艺参数显示成功或失败。
```

输入：

```text
glazeType：釉料
coverThickness：罩釉厚度
heatingSpeed：升温节奏
oxygenLevel：氧化气氛
```

输出：

```text
result.title
result.description
result.cause
result.suggestion
result.isSuccess
```

给 AI 的提示词：

```text
现在只实现一个功能：开窑结果判定。

已有参数：
- glazeType
- coverThickness
- heatingSpeed
- oxygenLevel

需求：
1. 根据参数返回一个 result 对象。
2. result 包含 title、description、cause、suggestion、isSuccess。
3. 失败结果包括颜色发灰、黑壳、针孔、气泡、发乌。
4. 成功条件是长石釉、薄罩釉、慢升温、氧化充分。
5. 不要改其他界面。
6. 保持现有代码结构。
7. 修改完成后说明改了哪些文件。
```

## 7. 第一周建议计划

### 第 1 天：写执行文档

- 完成垂直切片执行稿。
- 完成失败规则表初版。
- 完成界面清单。

### 第 2 天：做灰盒界面

- 开始界面。
- 工坊界面。
- 参数选择界面。
- 结果界面。

### 第 3 天：实现规则

- 参数保存。
- 开窑结果判定。
- 失败复盘文案。
- 再试一次。

### 第 4 天：补剧情

- 开场对白。
- 第一次失败后的角色提示。
- 成功后的展柜文案。

### 第 5 天：接入占位美术

- 工坊背景。
- 窑炉间背景。
- 成功瓷器图。
- 失败瓷器图。

### 第 6 天：试玩修正

- 修 bug。
- 缩短太长的文案。
- 调整失败提示。
- 确保能从头玩到尾。

### 第 7 天：整理版本

- 标记当前版本。
- 记录已完成内容。
- 写下一阶段待办。

## 8. 判断是否该继续扩展

在进入下一阶段前，先问 5 个问题：

1. 当前版本能否从头玩到尾？
2. 玩家是否知道下一步该做什么？
3. 失败是否有可理解原因？
4. 成功是否带来明确反馈？
5. 新增内容是否会破坏核心闭环？

如果有任意一个答案是否定的，先修当前版本，不要继续加系统。

## 9. 当前最推荐的下一步

从这里开始：

1. 新建 `重见五彩_垂直切片执行稿.md`。
2. 把 10-15 分钟流程写成逐分钟脚本。
3. 新建 `重见五彩_失败规则表.md`。
4. 把失败、原因、调整项写成表格。
5. 开始做纯文字按钮版灰盒原型。

第一目标不是“做完整游戏”，而是：

> 做出一个别人能独立玩完，并且玩完后理解三烧改二烧意义的小版本。

只要这个版本成立，后续章节、角色、美术、经营和图鉴都可以稳步加上去。
